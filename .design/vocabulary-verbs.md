# Feature: vocabulary declaration verbs — completing v0.2

## Summary

Declaring a telos or a process atom currently means hand-writing a fenced
JSON block inside a `kan observe` heredoc. That friction suppresses
recording, and suppressed recording undermines `telos/legible-process`
directly. This adds `day telos declare`, `day telos tension`, and `day atom
declare` over one shared vocabulary-writer, folds baseline setup into `day
init`, and adds the SessionEnd hook — completing v0.2.

## Requirements

- REQ-1: `day telos declare <slug> "<statement>"` appends a `decide` claim on
  `telos/<slug>`, optionally declaring the subject's title and kind
  (`--title`, `--kind`). If the subject already has live claims, the new
  claim **automatically cites the newest one**, so revision is the same verb
  as declaration — kan is append-only and a revision is just a later claim,
  so a separate `revise` verb would name an operation the data model does not
  have.
- REQ-2: `day telos tension <a> <b> "<why>"` appends a `decide` claim on `a`
  citing the newest live claim on both `telos/<a>` and `telos/<b>`. Tension
  between teloi is the relation that makes teloi more than a values list, and
  it is currently prose. This makes it a first-class verb whose output is
  still an ordinary citing claim, because kan has no "in tension with"
  `RelationKind` yet (tracked as a kan issue). When kan gains that variant,
  day emits it instead — the verb's surface does not change.
- REQ-3: `day atom declare <slug> --in <type>... --out <type>... --next
  <slug>...` appends an `observe` claim on `atom/<slug>` whose text carries a
  `day-atom` block **generated by day**, not hand-written. Same auto-citing
  revision behavior as REQ-1.
- REQ-4: `day atom declare` runs the composition check over the vocabulary
  *including the new atom*, prints any findings, and **records regardless**.
  Declaration-time feedback rather than after-the-fact discovery, but
  advisory: a vocabulary that does not yet compose is a normal intermediate
  state while you declare a chain of atoms one at a time, and refusing the
  write would make declaring a three-atom pipeline impossible in any order.
- REQ-5: `day init` performs baseline setup rather than only printing. It
  records the starter design-doc schema (`schema/design-doc`) if none is
  declared, and continues to **print, never write,** the harness and MCP
  wiring. The split is principled: day writes *claims* — append-only,
  attributable, retractable, and the user's own data — and never writes
  *config*, which is opaque, unattributable, and awkward to undo. `--print`
  suppresses all writing for anyone who wants the old behavior.
- REQ-6: `day init` is idempotent. With a schema already declared it reports
  that and records nothing; `--force` re-records deliberately.
- REQ-7: A `session-end` hook event reporting the subjects that are still
  open (via `kan issues`) and the declared teloi, and asking that outcomes,
  findings, and decisions be recorded before the context is lost. Advisory
  and exit-0 like every other hook, and registered in `hooks/hooks.json`
  alongside SessionStart.
- REQ-8: The hook reports what is **open**, not what changed "this session" —
  day has no store and therefore no session state, and acquiring one to
  answer that question would trade `telos/no-store-of-its-own` for a
  reminder. Open-subject state is derivable from the log alone and is the
  honest, stateless approximation.
- REQ-9: One shared vocabulary writer (`src/vocabulary.rs`) backing all three
  declaration verbs: resolve the subject, find the newest live claim to cite,
  build the claim text, append through `KanClient::append`. The verbs differ
  only in subject prefix, kan verb, and body — three thin surfaces over one
  mechanism, matching how `atoms` and `schema` already share
  `atoms::newest_fenced`.
- REQ-10: `docs/CONVENTIONS.md` documents that these verbs are the preferred
  way to declare vocabulary and that hand-written claims remain equally
  valid, since the conventions — not the tool — are the contract.

## Architecture

**`src/vocabulary.rs` (new)** is the shared writer: given a subject, a kan
verb, and claim text, find the newest live claim via `KanClient::show`, build
a `kan_client::Write` citing it when present, and append. Everything the
three verbs share lives here; nothing about teloi or atoms specifically does.

**`src/atoms.rs`** gains interface *serialization* to sit beside the
extraction it already has — `day atom declare` builds the `day-atom` block
that `extract_interface` reads, so the write and read paths share the
`Interface` type and cannot drift apart.

**`src/cli/mod.rs`** gains `telos` (`declare`, `tension`) and `atom`
(`declare`) subcommand groups, and `init` gains `--force`/`--print`. The
existing verbs are untouched.

**`src/hooks.rs`** gains `session_end`, dispatched by the existing
`hooks::dispatch` match, which already errors clearly on unknown events —
adding an arm is the whole change there.

**`src/schema.rs`** already knows `Schema::starter()`; `day init` records it
rather than printing it, so the starter has exactly one definition and the
printed and recorded forms cannot disagree.

**Nothing here destroys.** Every verb appends; revision is a later claim
citing an earlier one. day still writes only through kan's public CLI, keeps
no store, and cannot alter or destroy a subject.

## Resolved Questions

- **Tension verb ships now, as a citing claim**: `day telos tension` appends
  a `decide` citing both subjects rather than waiting for kan to grow an "in
  tension with" `RelationKind`. It works today, needs no kan release, and
  gets the most model-central relation out of prose; a kan issue tracks the
  variant so the follow-through is visible outside a claim in day's own log.
- **Declaration verbs only, no read verbs**: reading is already covered by
  kan's `show`/`status` plus day's `doctor` and `session_context`. Adding
  `list`/`show` would duplicate kan's surface for marginal gain against a
  standing house rule to keep day's CLI small. Revisit if reading proves
  awkward in practice rather than in anticipation.
- **No separate `revise` verb**: `declare` auto-cites the newest live claim
  when one exists. kan is append-only, so a revision *is* a later claim; a
  distinct verb would name an operation the data model does not have.
- **Baseline setup folds into `day init`** rather than a new `day schema`
  verb: init already exists for exactly this, and recording the starter
  schema is the one thing standing between a fresh repo and a working `day
  design check`. The write/print split — claims yes, config no — is what
  keeps this from being a silently-mutating setup command.
- **SessionEnd reports open subjects, not session deltas**: day has no store
  and therefore no session state, and acquiring one would trade
  `telos/no-store-of-its-own` for a reminder. What is open is derivable from
  the log alone.

## Out of Scope

- `day telos retract` or any retraction verb. Retraction is kan's, it is
  attributable, and day has no business wrapping it.
- Querying tensions as graph edges — blocked on kan's `RelationKind`, tracked
  as a kan issue, and not something day should work around locally.
- Frame conventions, still deliberately unconventionalized
  (`docs/CONVENTIONS.md`).
- The v0.3 bridging work: planning a path between teloi, work units, and
  parallel de-risking.
- Schema editing field-by-field. Schemas change rarely; `day init` records a
  starter and revising one is an ordinary kan claim.
- Any change to kan.
